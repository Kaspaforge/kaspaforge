# Kaspa Forge Deal Recovery Kit — `dealctl`

**`dealctl`** — публичный CLI восстановления на стороне участника для сделок Kaspa Forge **Гарант**
(Escrow) и **Залог** (Deposit). Его единственная задача — **заново собрать и подписать любой путь
ковенанта, который уже разрешён вашим собственным ключом**, используя только локальную recovery-запись
и Kaspa-ноду по вашему выбору.

> **Ни на одном шаге не происходит обращения к серверу Kaspa Forge.** Ни чтобы найти вашу сделку, ни
> чтобы построить транзакцию, ни чтобы её отправить. Инструмент общается только с той Kaspa-нодой, на
> которую вы его нацелите.

Этот набор существует ради принципа: **«смерть сервиса ≠ потеря денег».** Даже если `kaspaforge.org`
и весь hosted-бэкенд навсегда недоступны, вы всё равно можете выполнить `release`, `refund`,
`dispute`, keyless-пути `auto-release`/`timeout` и урегулировать сделку по взаимной подписи — целиком
на своей машине.

## Чего он НЕ умеет

`dealctl` лишь **делает доступными ончейн-права, которыми вы уже обладаете**. Он честен о своих
границах:

- Он **не может** восстановить утерянный пароль от `.age`-backup.
- Он **не может** восстановить утерянный приватный ключ стороны.
- Он **не может** обойти контракт эскроу или выдать вам путь, который ваш ключ не авторизует.
- Он **не может** сделать так, чтобы таймлок созрел раньше срока — keyless-путь
  `auto-release`/`timeout` сработает только после того, как ончейн-возраст в DAA перешагнёт окно
  контракта.

Если у вас есть ключ и таймлок созрел — право ваше, и этот инструмент даёт вам его реализовать. Если
нет — ни один инструмент, ни наш, ни чей-либо ещё, не сможет его создать.

## 1. Проверьте то, что скачали

Собирайте из публичного исходника и сверяйте с опубликованным манифестом контрольных сумм. **Не**
запускайте бинарник из неофициального зеркала.

```bash
# Клонируйте публичный репозиторий с исходниками
git clone https://github.com/Kaspaforge/kaspaforge
cd kaspaforge/recovery-kit

# Зафиксируйте точный коммит, который собираете
git log -1 --pretty=oneline

# Сверьте каждый поставляемый файл с подписанным манифестом контрольных сумм
sha256sum -c RECOVERY-SHA256SUMS
```

`RECOVERY-SHA256SUMS` покрывает ядро контракта, CLI, JSON-схемы, тестовые векторы и офлайн-дешифратор
`.age`. Каждая строка должна показать `OK`. Если хоть что-то не сошлось — остановитесь.

## 2. Соберите из исходников

```bash
cargo build --release -p dealctl
```

Бинарник появится в `target/release/dealctl`. Сборка из исходников — канонический путь; готовый
release-бинарник этот шаг не заменяет — контрольные суммы проверяйте в любом случае.

```bash
./target/release/dealctl help
```

## 3. Расшифруйте свой `.age`-backup — офлайн

Ваш Desk-backup — это файл, зашифрованный `age` по парольной фразе. Расшифруйте его на **офлайн**-машине,
чтобы получить `profile.json`. Два равнозначных способа:

**Вариант A — офлайн-инструмент в браузере.** Откройте `keyfile-decrypt.html` из набора (одна
статическая страница, без сети) в браузере, загрузите свой `.age`-файл, введите парольную фразу и
сохраните расшифрованный `profile.json`.

**Вариант B — CLI `age`.**

```bash
age --decrypt -o profile.json your-backup.age
# age запросит вашу парольную фразу
```

`profile.json` — это открытый текст, содержащий приватные ключи. Держите его на офлайн-машине и
удалите, когда закончите (см. шаг 9).

## 4. Извлеките минимальную recovery-запись

Никогда не передавайте денежным командам полный `profile.json`. Сначала извлеките минимальную запись
по одной сделке:

```bash
dealctl extract \
  --profile profile.json \
  --deal <deal-id> \
  --output deal.recovery.json
```

- Файл создаётся с правами `0600` и **никогда не перезаписывается**: если `deal.recovery.json` уже
  существует, `extract` откажется, а не затрёт его.
- Запись хранит только то, что нужно этой конкретной сделке: вашу роль, ключи, параметры контракта,
  fingerprint и адреса ACTIVE/DISPUTED. Она намеренно исключает любые service-токены, ключ чата, seed
  и другие сделки.

## 5. Проверьте запись — без сети

```bash
dealctl verify --recovery deal.recovery.json
```

`verify` работает полностью **офлайн**. Он проверяет схему и версию контракта, подтверждает, что
fingerprint исходника контракта поддерживается, пересчитывает ваш публичный ключ из приватного,
сверяет, что ключ соответствует заявленной роли, и локально заново выводит адреса ACTIVE и DISPUTED,
чтобы убедиться в их совпадении с записью. Если любая проверка не прошла — инструмент останавливается
ещё до выхода в сеть.

## 6. Посмотрите ончейн-статус — через свою ноду

```bash
dealctl status --recovery deal.recovery.json --node grpc://127.0.0.1:16110
# добавьте --json для машиночитаемого вывода
```

`status` запрашивает у ноды адреса ACTIVE и DISPUTED сделки и перечисляет **каждый** найденный
covenant-UTXO — по строке на каждый — показывая:

- **режим** (ACTIVE или DISPUTED),
- **outpoint** (`txid:index`),
- **сумму**,
- **covenant id**,
- **возраст в DAA** и сколько DAA осталось до момента, когда станут возможны
  `auto-release`/`timeout`,
- **действия, доступные вашей роли** прямо сейчас.

Нода рассматривается как **недоверенный ввод**: `status` никогда не позволяет ноде изменить, куда
уходят деньги. Все адреса и скрипты вычисляются локально из вашей recovery-записи; нода лишь сообщает,
какие UTXO существуют. Нода по умолчанию — loopback (`grpc://127.0.0.1:16110`); удалённая нода
задаётся явно через `--node grpc://your-node:16110`.

### Одна транзакция на один UTXO

`dealctl` **никогда не объединяет два covenant-UTXO в один spend.** Если на адресе сделки лежит больше
одного UTXO, денежной команде нужно указать, какой именно:

- `--outpoint txid:index` действует ровно на этот UTXO, либо
- `--all` строит **отдельную** транзакцию для **каждого** подходящего UTXO.

Если подходящий UTXO ровно один, оба флага можно опустить.

## 7. Таблицы прав — кто что может

Денежная команда сработает, только если ваш ключ её авторизует, а для keyless-путей — если таймлок
созрел. `dealctl` отказывается подписывать ключом не той роли даже в случаях, которые сам контракт бы
отклонил.

### Гарант (Escrow)

```bash
dealctl escrow release   --recovery deal.recovery.json [--node …] [--outpoint txid:index | --all] [--output tx.json] [--broadcast]
dealctl escrow refund    --recovery deal.recovery.json …
dealctl escrow dispute   --recovery deal.recovery.json …
dealctl escrow auto-release --recovery deal.recovery.json …
dealctl escrow timeout   --recovery deal.recovery.json …
```

| Команда        | Режим                            | Авторизация | Результат                                    |
|----------------|----------------------------------|-------------|----------------------------------------------|
| `release`      | ACTIVE или DISPUTED              | ключ buyer  | всё **продавцу (seller)** + resolve-комиссия |
| `refund`       | ACTIVE или DISPUTED              | ключ seller | всё **покупателю (buyer)** + resolve-комиссия |
| `dispute`      | ACTIVE                           | ключ buyer  | перевод в **DISPUTED** (тот же ковенант)     |
| `auto-release` | ACTIVE, возраст ≥ dispute_window | **без ключа** | **продавцу (seller)** + resolve-комиссия    |
| `timeout`      | DISPUTED, возраст ≥ arbiter_deadline | **без ключа** | стороне **`timeout_to`**, **без service-комиссии** |

### Залог (Deposit)

Залог работает на том же контракте с переименованными ролями. **Держатель (holder)** — это buyer
контракта; **вкладчик (depositor)** — это seller контракта. В recovery-записи должно стоять
`product = deposit` — применить терминологию Залога к обычной Escrow-записи по ошибке нельзя.

```bash
dealctl deposit return      --recovery deal.recovery.json …
dealctl deposit concede     --recovery deal.recovery.json …
dealctl deposit claim       --recovery deal.recovery.json …
dealctl deposit auto-return --recovery deal.recovery.json …
dealctl deposit timeout     --recovery deal.recovery.json …
```

| Команда       | Соответствует  | Режим                                | Авторизация    | Результат                                          |
|---------------|----------------|--------------------------------------|----------------|----------------------------------------------------|
| `return`      | `release`      | ACTIVE или DISPUTED                  | ключ holder    | всё **вкладчику (depositor)** + resolve-комиссия   |
| `concede`     | `refund`       | ACTIVE или DISPUTED                  | ключ depositor | всё **держателю (holder)** + resolve-комиссия      |
| `claim`       | `dispute`      | ACTIVE                               | ключ holder    | перевод в **DISPUTED** (тот же ковенант)           |
| `auto-return` | `auto-release` | ACTIVE, возраст ≥ dispute_window     | **без ключа**  | **вкладчику (depositor)** + resolve-комиссия       |
| `timeout`     | `timeout`      | DISPUTED, возраст ≥ arbiter_deadline | **без ключа**  | стороне **`timeout_to`**, **без service-комиссии** |

### Восстановление фондирования

Если фондирование сделки было прервано, эти команды его завершают или откатывают. Обе требуют
проверенного funding-ключа в записи.

```bash
# Завершить прерванную закладку: перевести funding-UTXO в ACTIVE covenant genesis
dealctl fund --recovery deal.recovery.json [--node …] [--output tx.json] [--broadcast]

# Вернуть ещё не залоченные средства на обычный адрес под вашим контролем
dealctl sweep-funding --recovery deal.recovery.json --to kaspa:q… [--node …] [--broadcast]
```

### Взаимное урегулирование

Две стороны могут урегулировать сделку, совместно подписав одну согласованную транзакцию, без
арбитра. Каждый подписывает свою половину; затем любая из сторон объединяет обе подписи.

```bash
# Каждая сторона подписывает, согласовав сумму покупателю (в sompi)
dealctl mutual-sign --recovery deal.recovery.json --to-buyer <sompi> [--outpoint txid:index] --output me.sig.json

# Объединить две подписи в итоговую транзакцию
dealctl mutual-combine --recovery deal.recovery.json \
  --buyer-signature buyer.sig.json \
  --seller-signature seller.sig.json \
  [--output tx.json] [--broadcast]
```

`mutual-combine` принимает только две подписи от **разных** ролей над **в точности одной и той же**
транзакцией. Любое расхождение в параметрах, выходах, режиме или outpoint его останавливает.

## 8. Подпись офлайн, отправка онлайн

Каждая денежная команда следует одному безопасному правилу по умолчанию:

1. строит транзакцию **офлайн**,
2. прогоняет **локальный верификатор** по её входам и выходам,
3. печатает **человекочитаемый preview**,
4. сохраняет в `--output`, **если** вы указали этот флаг,
5. и **не отправляет** ничего, пока вы не передадите `--broadcast`.

### Подпись на air-gapped машине (ключ не выходит в сеть)

Чтобы подписывать приватным ключом на машине **без сети вообще**, используйте четырёхшаговый флоу. Ноду
трогают только первый и последний шаги; подпись — полностью офлайн.

Приватный ключ **никогда не попадает на онлайн-машину** — онлайн-шаги работают с публичным watch-файлом.

```bash
# 1) ОФЛАЙН — записать ПУБЛИЧНЫЙ watch-файл сделки (только адреса и id, БЕЗ приватного ключа):
dealctl watch --recovery deal.recovery.json --output watch.json

# 2) Перенести watch.json на ОНЛАЙН-машину и выгрузить covenant-линии (ключа на этой машине НЕТ):
dealctl prepare --watch watch.json --node grpc://your-node:16110 --output lines.json

# 3) Перенести lines.json ОБРАТНО на ОФЛАЙН-машину и ПОДПИСАТЬ — без --node:
dealctl escrow release --recovery deal.recovery.json --line lines.json --output release.tx.json
#   (если линий несколько, добавьте --outpoint txid:index, чтобы выбрать одну)

# 4) Перенести release.tx.json на ОНЛАЙН-машину и отправить — приватный ключ НЕ нужен:
dealctl submit --tx release.tx.json --node grpc://your-node:16110
```

На онлайн-машину попадают только `watch.json`, `lines.json` и `release.tx.json` — ни в одном нет
приватного ключа. Шаг 3 целиком офлайн и **привязывает** `lines.json` к вашей recovery-записи (network,
product и deal_id должны совпасть), поэтому пакет, подготовленный для другой сделки, подписать нельзя.
`submit` заново проверяет конверт, txid и структурную политику пути перед отправкой.

> Продвинуто: вместо `--line lines.json` можно задать линию вручную всеми четырьмя флагами
> `--outpoint txid:index --amount <sompi> --covenant-id <hex> --mode 0|1`. Денежная команда подключается
> к ноде ТОЛЬКО если не передан ни `--line`, ни этот полный набор.

Прежде чем что-либо отправлять, **прочитайте preview**: проверьте сеть, режим/outpoint, каждый
адрес и сумму на выходе и комиссии. Инструмент строит каждый показанный адрес из проверенного
output-скрипта, но окончательная ответственность убедиться, что деньги уходят туда, куда вы ожидаете,
лежит на вас.

## 9. Уберите за собой

`profile.json` и промежуточные файлы содержат приватные ключи или материал, несущий ключи. Закончив,
удалите их с машины:

```bash
rm -f profile.json deal.recovery.json me.sig.json *.sig.json
# оставьте только то, что вы осознанно хотите сохранить
```

Если сделка ещё за вами, храните исходный `.age`-backup в безопасности и офлайн — именно он позволит
вам заново получить `profile.json` в будущем.

## 10. Честные ограничения

- Набор восстанавливает **только** те сделки, полную recovery-запись которых вы уже держите. Он не
  может реконструировать redeem-script сделки из одного лишь приватного ключа — параметры конструктора
  должны быть в вашем backup. Делайте свежий `.age`-backup после фондирования каждой сделки.
- Он **не** ищет контрагента, **не** судит о том, была ли сделка исполнена, и **не** ведёт арбитраж.
  Он лишь строит ончейн-пути, которые контракт уже разрешает.
- Он не запускает чат, медиатора, маркетплейс, registry или любую hosted-функцию. Всё это по замыслу
  вне scope и не входит в этот пакет.
- Кворум из нескольких нод не поддерживается; используется одна выбранная вами нода, и она считается
  недоверенной.

## ⚠️ Обязательные предупреждения

- **Утерянный пароль от `.age` восстановить нельзя.** Ни бэкдора, ни сброса не существует.
- **Утерянный приватный ключ стороны восстановить нельзя.** Без него пути, которые он авторизует,
  потеряны.
- **Для ончейн-восстановления service-токен не нужен.** Всё здесь работает от одних ваших ключей и
  ноды.
- **Поддержка НИКОГДА не попросит ваш `.age`-файл, пароль, profile/recovery JSON или приватный ключ.**
  Кто просит — тот на вас нападает. Не отправляйте их.
- **Не устанавливайте бинарники из неофициальных источников.** Собирайте из исходников или сверяйте
  опубликованные контрольные суммы, прежде чем что-либо запускать.
- **`--all` создаёт несколько отдельных транзакций** — по одной на UTXO. Знайте, на сколько UTXO вы
  действуете, прежде чем его использовать.
- **Перед отправкой всегда проверяйте адреса и суммы в preview.** Транзакцию, уже попавшую в сеть,
  отменить нельзя.
